# API Service Planning

This document defines how to identify, scope, and organize service boundaries for FoundryForge projects. A service is a module that owns a specific domain of business logic and communicates with external systems or databases. Proper service boundaries prevent circular dependencies, keep files testable, and allow teams to work on separate domains without constant merge conflicts.

Every Feature Specification generated by FoundryForge should include a Services section that enumerates the required services, their responsibilities, and their interaction patterns.

## What Makes a Good Service Boundary

A service should own one cohesive area of responsibility. The classic test for cohesion is to describe what a service does in a single sentence without using "and" or "or." If the sentence requires conjunctions, the service is doing too much.

Another boundary test is the rate-of-change heuristic. If two pieces of logic change for different reasons, they belong in different services. Authentication logic changes when security requirements evolve, while payment logic changes when pricing or billing models shift. They may depend on each other, but they should not live in the same file.

Services map roughly to bounded contexts in domain-driven design. Every service owns its data model, its validation rules, and its external integration for that domain. One service should not reach into another service's database tables — it should call the owning service's public methods instead.

## Service Catalog

The following service types appear in most FoundryForge projects. Not every project needs every service, but every project should document which services are in scope and which are explicitly out of scope.

### Auth Service

The Auth Service manages identity, authentication, and authorization primitives. It confirms that a user is who they claim to be and determines what resources they can access.

Responsibilities include user registration and sign-in, password hashing and verification, session management via JWT or cookies, OAuth provider integration (Google, GitHub, Apple), multi-factor authentication enrollment and verification, role and permission assignment, and token refresh and revocation.

The Auth Service must never handle business data such as user profiles, payment history, or preferences. Its data model is limited to credentials, sessions, and roles. User profile data belongs to the User Service.

Auth Service interactions follow a request-verification pattern. Other services call `authService.verifySession(token)` to get the current user identity, but they never call auth methods to check permissions. Permission checks use a separate authorization function that combines the user's roles with the resource's access policy.

```
// Auth Service provides identity verification
const session = await authService.verifySession(request.headers.authorization);
// Returns { userId, roles: ['admin', 'member'] } or throws UnauthorizedError

// Authorization is a separate concern
authorize(session.roles, 'resource:write');
```

### User Service

The User Service owns the user profile domain. It manages personal information, account settings, and user preferences.

Responsibilities include profile CRUD operations (name, avatar, bio), account settings (language, timezone, notification preferences), user search and discovery (by name, email, or attributes), account deletion and data export for privacy compliance (GDPR, CCPA), and admin operations like suspending or banning accounts.

The User Service must not contain authentication logic. Resetting a password requires coordination between User Service and Auth Service, but the password hash itself belongs to Auth. The User Service calls `authService.initiatePasswordReset(email)` when a user requests a password change through their profile settings.

User Service data includes a `users` table with mutable profile fields and a separate `user_preferences` table for settings. The service should cache frequently accessed profile data to reduce database load on high-traffic endpoints.

### Payment Service

The Payment Service handles monetary transactions, subscriptions, invoices, and billing. It integrates with payment providers such as Stripe, Lemon Squeezy, or Paddle.

Responsibilities include creating and managing checkout sessions (one-time and recurring), handling payment method attachments and updates, subscription lifecycle management (create, upgrade, downgrade, cancel), invoice generation and payment reconciliation, refund processing, webhook handling for payment provider events, and billing address and tax collection.

The Payment Service must be designed for asynchronous processing. Payment confirmations arrive via webhooks, not synchronous API responses. A checkout session creation call returns a URL for the user to complete payment on the provider's checkout page. When the provider sends a `checkout.session.completed` webhook, the Payment Service updates the subscription status and calls the Notification Service to send a confirmation email.

```
// Payment Service always works asynchronously via webhooks
async function createCheckoutSession(userId: string, priceId: string): Promise<string> {
  const session = await stripe.checkout.sessions.create({
    customer: await this.getOrCreateCustomer(userId),
    line_items: [{ price: priceId, quantity: 1 }],
    mode: 'subscription',
    success_url: `${FRONTEND_URL}/dashboard`,
    cancel_url: `${FRONTEND_URL}/pricing`,
  });
  await this.savePendingSession(userId, session.id);
  return session.url;
}

// Webhook handler — this is where the real logic lives
async function handleCheckoutCompleted(event: Stripe.Event): Promise<void> {
  const session = event.data.object as Stripe.Checkout.Session;
  const userId = await this.lookupUserIdByCustomerId(session.customer as string);
  await this.activateSubscription(userId, session.subscription as string);
  await notificationService.sendPaymentConfirmation(userId);
}
```

A common mistake is putting subscription entitlement checks (does this user have access to feature X?) into the Payment Service. Entitlement logic belongs in a dedicated Entitlement Service or in the feature-flag system. The Payment Service reports the subscription plan and status; it does not evaluate feature access.

### Notification Service

The Notification Service abstracts the delivery of messages to users across multiple channels: email, push notifications, in-app toasts, and SMS.

Responsibilities include email delivery via providers like SendGrid, Resend, or SES, push notification delivery to mobile and browser targets, in-app notification storage and retrieval, SMS delivery via Twilio or Vonage, notification preference management (users choose which channels they want for each event type), and notification templating with dynamic variable substitution.

The Notification Service should be called by other services but should never initiate notifications on its own. The Payment Service calls `notificationService.sendPaymentConfirmation(userId)` after processing a webhook. The Notification Service does not know about payment logic; it only knows that it received a user ID and an event type, and it looks up the user's preferences and sends accordingly.

```
// Other services tell the Notification Service what happened
await notificationService.send({
  userId: payment.userId,
  eventType: 'payment.confirmed',
  data: { amount: payment.amount, date: payment.createdAt },
});
// The Notification Service handles channel selection and templating internally
```

Notification templates must be stored in a database or configuration file, not hardcoded in service code. This allows non-developers to update email copy without a deployment.

### Search Service

The Search Service provides full-text search capabilities across application entities. It owns the search index and query logic.

Responsibilities include indexing entities (users, documents, products) into a search engine such as Meilisearch, Algolia, or Typesense, handling search queries with faceting, filtering, and sorting, managing index synchronization when source data changes (via webhooks or scheduled jobs), and providing autocomplete and typo-tolerant search results.

The Search Service is a read-optimized view of data owned by other services. When the User Service updates a user's name, it publishes an event that the Search Service consumes to update its index. The Search Service never writes to primary databases.

```
// User Service publishes an event after update
await eventBus.publish('user.updated', { userId, name, email });
// Search Service consumes the event
eventBus.subscribe('user.updated', async (event) => {
  await searchClient.index('users').updateDocument({
    id: event.userId,
    name: event.name,
    email: event.email,
  });
});
```

Search queries return document IDs and snippets. The calling service or API layer uses those IDs to fetch full documents from the appropriate service. Returning full documents from the search index leads to stale data and sync complexity.

### Analytics Service

The Analytics Service tracks user behavior and product events for business intelligence and product improvement.

Responsibilities include event ingestion from client and server sources, session recording and replay (if applicable), funnel and retention analysis data preparation, dashboard data aggregation and precomputation, and export pipelines for data warehouses (BigQuery, Snowflake).

Client-side events should be sent to the Analytics Service asynchronously with no impact on user experience. The analytics endpoint accepts batches of events with minimal validation — dropping an analytics event should never cause an application error.

The Analytics Service must respect user privacy preferences. Before recording an event, it checks the user's consent status stored in the User Service. Events for non-consenting users are anonymized or dropped entirely.

### Storage Service

The Storage Service manages file uploads, downloads, and asset lifecycle. It abstracts the underlying blob storage provider (S3, GCS, Firebase Storage, R2).

Responsibilities include generating signed upload URLs for direct client-to-storage uploads, serving files with appropriate content types and cache headers, managing file deletion and retention policies, image transformation (resize, crop, format conversion) via integration with image processing services, and enforcing file type and size restrictions.

The Storage Service should never handle user identity or access control beyond verifying that the requester has a valid session. File-level access control (e.g., "only the document owner can download this file") belongs in the domain service that owns the document, not in the Storage Service.

The Storage Service uses a path convention to organize files:

```
uploads/{tenantId}/{entityType}/{entityId}/{filename}
```

This structure keeps files organized without requiring a database for folder listing, and it naturally supports multi-tenant isolation.

## Service Interaction Patterns

Services should communicate through well-defined interfaces, not through shared databases or direct imports of each other's internals.

### Synchronous Communication (Direct Service Calls)

Use synchronous calls when the calling service needs an immediate result to complete its operation. A controller that needs the current user's profile calls `userService.getProfile(userId)` synchronously because it cannot render the response without the data.

Services must never call each other in a deep chain within a single request. If creating a user requires creating a Stripe customer, sending a welcome email, and initializing analytics tracking, the controller or use case orchestrator should call each service separately, not chain them through User Service → Payment Service → Notification Service.

```
// Do: orchestrate at the controller layer
async function signUp(req: Request, res: Response) {
  const user = await userService.create(req.body);
  const customer = await paymentService.createCustomer(user.id, user.email);
  await notificationService.sendWelcomeEmail(user.email);
  await analyticsService.trackEvent('user.signed_up', { userId: user.id });
  res.status(201).json({ user, customer });
}

// Don't: chain services internally
// userService.create() should NOT internally call paymentService.createCustomer()
```

### Asynchronous Communication (Events)

Use events when the caller does not need an immediate response or when the operation involves multiple downstream services that can run independently. User registration triggers welcome emails, analytics tracking, and search index updates — all of which can happen asynchronously without delaying the sign-up response.

Events should follow a `{domain}.{action}` naming convention: `user.created`, `payment.completed`, `subscription.canceled`. Each event carries a payload with the entity ID and relevant data, never the full entity state.

The event bus can be implemented with a message queue (RabbitMQ, Redis Pub/Sub, Google Pub/Sub) or a database-backed outbox pattern. For small to medium projects, the outbox pattern suffices: the service writes an event to an `outbox` table within the same database transaction as the data change, and a separate process reads and publishes those events.

## When to Split a Service

Split a service when it accumulates responsibilities that change for different reasons. If the Auth Service's sign-up logic keeps changing because marketing wants different onboarding flows, but its token generation logic is stable, those two concerns may need separate files within the service. If they also need different database access patterns or have different scaling requirements (e.g., token verification needs sub-millisecond latency while onboarding flows do not), they warrant separate services.

Split a service when its data model grows beyond five or six tables that do not share a common root. A sprawling service with twenty tables that have no clear relationship is a bounded context that has been misidentified.

Split a service when two teams need to deploy changes to it independently of each other. If the frontend team and the mobile team both need different changes to the User Service this week, the User Service may need to be decomposed into a user-profile service and a user-settings service.

## When to Merge Services

Merge services when they always change together. If every change to the Payment Service requires a corresponding change to the Entitlement Service, and they are always deployed as a pair, they should be a single service.

Merge services when they perform a database join on every query. If the Auth Service and User Service are joined on every request and never deployed independently, the split is adding complexity without benefit.

Merge services when event-driven synchronization causes noticeable staleness. If the Search Service cannot serve results because its index is perpetually behind the source data, merging the search indexing logic into the source service (and using database-level search features) may be the pragmatic choice.

## Do's and Don'ts

**Do** return domain objects from service methods, not raw database records. A service should map database rows into typed domain objects before returning them to callers. This insulates callers from schema changes.

**Don't** let services import from other services' repository or database files. All cross-service communication must go through the service's public methods. A repository is an implementation detail of its owning service.

**Do** write integration tests for every public service method. Unit tests with mocked dependencies are useful for complex logic, but integration tests that exercise the actual database and external integrations catch the real bugs.

**Don't** put HTTP request or response handling in services. Services receive plain data and return plain data. Controllers and route handlers translate between HTTP and service methods.

**Do** design service methods to be idempotent when they have side effects. If a payment webhook is retried, calling `activateSubscription` again with the same subscription ID should be safe. Use database constraints or idempotency keys to enforce this.
